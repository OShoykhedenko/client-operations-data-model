    Шаг 1.
Я сделал первоночальную реализацию дата модели. Сделал базовые схемы БД, описанные в ТЗ. 
Я решил сделать статусы БД клиентов и договоров BOOLEAN, потому что в ТЗ у них бинарный статус (активный и закрытый). 
Так же я сделал ENUM для типов договоров и для валюты, чтобы ограничить допустимые значения без усложнения модели. На прод версии я бы сделал dictionaries для типов договоров, чтобы добавить масштабироваемость.
Также я сделал индексы под ключевые joins и фильтрации операций по датам, чтобы запросы в ТЗ работали эффективнее. 

    Шаг 2. 
Нормализация данных.
Я решил изменить boolean is_active статусы на dictionaries (client_status_dict и contract_status_dict). Я подумал, что в реальных системах статус редко ограничивается двумя состояниями. Справочник позволит добовлять новые статусы без изменения схемы. Это сделает модель более гибкой и устойчивой к изменениям бизнес логики. Также я добавил флаг is_terminal, я подумал, будет логично добавить статус, по которому видно, что объект больше меняться не будет. Например, если статус активный, то is_terminal = false, а если статус closed, то is_terminal = true. Я думаю, это может упростить логику поиска, и вместо запроса WHERE status IN ('closed','archived','cancelled') можно просто использовать запрос WHERE is_terminal = true.
Также я решил изменить ENUM на dictionaries contract_type_dict и currency_dict. Это улучшит масштабируемость без изменения схемы.
Из не больших изменений: я решил использовать smallint вмето текстовых значений для direction in operations (было TEXT NOT NULL CHECK (direction IN ('Дт','Кт')), стало SMALLINT CHECK (direction IN (1,2))). Я считаю, что это будет более эффективно при больших объемах данных и уменьшит риск ошибок при написании sql запросов. 
Также я добавил новую таблицу accounts. Я подумал, что использовать аккаунт ид не совсем правильно для прода, потому что у людей может быть много разных аккаутов. Я думаю, такая модель ближе к реальным банковским системам. Также я добавил seed.sql с тестовыми данными для демонстрации работы модели и проверки запросов. Данные сгенерированы с помощью AI-помощьника и адаптированы для того, чтобы задачи в ТЗ гарантированно возвращали результаты и могли использоваться для локальных тестов.

    Шаг 4. 
Третий шаг правильно работает для маленьких объемов данных, но при миллионах транзакций в день за 30 дней накопятся десятки или даже сотни миллионов строк. Поэтому агрегация SUM raw-table operations станет очень дорогой операцией. 
Первый делом, для оптимизации, я добавил использование индексов для таблицы operations -> operations(account_id, created_at). Такой индекс позволит БД быстро находить нужные счета в правильном временном диапазоне, вместо того что бы сканировать всю таблицу. 
Хоть это и оптимизирует работу, все равно остается необходимость агрегировать большие объемы данных. Поэтому следующим этапом я бы добавил предагрегацию (daily agregations), что позволит занчительно уменьшить объем данных в обработке запроса.
Для прод версии я бы добавил промежуточную агрегированную таблицу, и она должна хранить агрегированные значения по каждому дню и счету. Агрегаты должны обновляться инкементально через ETL piplines. Это уменьшит объем запроса на порядки и сделает выполнение запроса намного бытсрее. В проде такой pipeline я бы реализовал через Airflow DAG, который будет: 
    1. агрегировать операции за новый день
    2. обновлять таблицу агрегатов
    3. использовать эту таблицу для аналитических витрин
